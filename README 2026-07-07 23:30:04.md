# Databricks End-to-End Flights Project

An end-to-end data engineering pipeline on Databricks that ingests raw flight/booking data, processes it through a Medallion Architecture (Bronze → Silver → Gold), and models it as a star schema (fact and dimension tables) for analytics.

## Architecture Overview

```
Raw CSV Files → Bronze Layer → Silver Layer → Gold Layer
                (Job/Workflow)  (DLT Pipeline)  (Dimensions + Fact)
```

| Layer | Purpose | Tooling |
|---|---|---|
| **Bronze** | Raw ingestion, minimal transformation | Databricks Job/Workflow (parameterized, loops over sources) |
| **Silver** | Cleaned, conformed, deduplicated data | Delta Live Tables (DLT) pipeline |
| **Gold** | Business-ready star schema | PySpark notebooks — Dimension (SCD Type 1) and Fact table builds |

## Source Data

Raw CSV files representing an airline booking system, including full-load and incremental (delta) variants:

- `dim_airports.csv` / `dim_airports_increment.csv`
- `dim_flights.csv` / `dim_flights_increment.csv`
- `dim_passengers.csv` / `dim_passengers_increment.csv` / `dim_passengers_scd.csv`
- `fact_bookings.csv` / `fact_bookings_increment.csv`

The `_increment` files simulate new/changed records for testing incremental (CDC-based) loads. `dim_passengers_scd.csv` simulates a Slowly Changing Dimension scenario for passenger attribute updates.

## Bronze Layer — Ingestion

**Type:** Databricks Job (Workflow)

The Bronze job ingests raw source files into Delta tables with minimal transformation, preserving raw structure for downstream processing.

**Job structure (`BronzeIngestion`):**
1. **`parameters` task** — runs a setup notebook that determines which sources need to be ingested and outputs a list of source configs (e.g., table names, file paths).
2. **`IncrementalIngestion` task (`for_each_task`)** — loops over the list produced by the `parameters` task, running the `BronzeLayer` notebook once per source, ingesting each file/table into its corresponding Bronze Delta table.

This pattern allows adding new source files by updating the parameters list, without duplicating ingestion logic per source.

The job's notebook source is configured via **Git integration** (`git_source`), pulling `source parameters` and `BronzeLayer` notebooks directly from this repository's `main` branch on each run.

## Silver Layer — Transformation

**Type:** Delta Live Tables (DLT) Pipeline

The Silver layer applies data quality rules, deduplication, type casting, and conforms Bronze data into clean, standardized tables (e.g., `silver_passengers`, `silver_bookings`), each carrying a `modifiedDate` column used as the Change Data Capture (CDC) watermark for downstream incremental processing.

## Gold Layer — Dimensional Model

**Type:** PySpark notebooks, organized as Dimension builds and a Fact build

### Dimensions (SCD Type 1)

Each dimension table (e.g., `DIMpassengers`, `DimAirports`, `DimFlights`) is built with:

- **Incremental CDC filtering** — pulls only Silver records where `modifiedDate >= last_load`, where `last_load` is derived from `MAX(updated_date)` in the existing Gold table (or a configurable `backdated_refresh` for historical reloads).
- **Surrogate key generation** — sequential integer keys generated via `row_number()` over a global window, continuing from the current `MAX(surrogate_key)` in the target table (avoids Spark's non-sequential `monotonically_increasing_id()`).
- **Audit columns:**
  - `created_date` — set once, on first insert
  - `updated_date` — refreshed on every insert/update; also serves as the incremental watermark for the next run
  - `modifiedDate` — carried through from Silver, representing when the source record itself changed (lineage)
- **Upsert via Delta `MERGE`** — matched records are updated, unmatched records are inserted, keyed on business key columns (e.g., `passenger_id`).

### Fact Table (`factBookings`)

Built by dynamically joining the Silver fact source (`silver_bookings`) against all relevant dimension tables to resolve surrogate keys:

- A Python query generator (`generate_fact_query_incremental`) builds the `SELECT` / `LEFT JOIN` / `WHERE` SQL dynamically based on a configurable list of dimensions and their join keys, avoiding hardcoded joins.
- Incremental filtering uses the same CDC pattern (`modifiedDate >= last_load`) against the Silver source.
- Loaded via Delta `MERGE`, matched on fact grain keys, with `whenMatchedUpdateAll()` / `whenNotMatchedInsertAll()`.

## Deployment

The Bronze ingestion job is defined as a **Databricks Asset Bundle**, allowing it to be version-controlled and deployed via the Databricks CLI:

```bash
databricks bundle deploy -t dev
```

This creates/updates the `BronzeIngestion` job in the target workspace using the definition in `resources/bronze_ingestion_job.yml`, with notebook source pulled live from this repository's `main` branch.

## Incremental Load Pattern (used across Bronze/Silver/Gold)

1. Determine `last_load` — the latest processed timestamp, read from the target table (or a manual `backdated_refresh` override for reprocessing).
2. Filter the source using `WHERE {cdc_col} >= last_load`.
3. Transform/enrich as needed for the given layer.
4. `MERGE` into the target table — update matched records, insert new ones.
5. `updated_date` on the target table naturally becomes the watermark for the next run.

## Notes

- Surrogate keys are generated using `row_number()` over an unpartitioned window to guarantee clean, sequential values — a deliberate tradeoff of parallelism (single partition during key assignment) for correctness. Tables are repartitioned after key assignment, before writing, to restore write parallelism.
- `key_cols`, `cdc_col`, and other configuration values are parameterized at the top of each notebook for reuse across tables.
